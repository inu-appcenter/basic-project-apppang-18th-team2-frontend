# 개발 컨벤션

## 📝 커밋 컨벤션

### 커밋 메시지 형식

```
<emoji> <commit type> : 설명
```

### 이모지 타입

| 이모지 | 타입 | 설명 | 예시 |
| --- | --- | --- | --- |
| ✨ | feat | 새로운 기능 추가 | ✨ feat : 영상 분석 결과 PDF 다운로드 기능 추가 |
| 🐛 | fix | 버그 수정 | 🐛 fix : 영상 업로드 시 발생하는 500 에러 수정 |
| 📝 | docs | 문서 변경 | 📝 docs : API 문서 업데이트 |
| 💄 | style | 코드 스타일 변경 (포맷팅, 세미콜론 등) | 💄 style : ESLint 규칙에 맞게 코드 포맷팅 |
| ♻️ | refactor | 코드 리팩토링 | ♻️ refactor : 영상 분석 로직 모듈화 |
| ⚡ | perf | 성능 개선 | ⚡ perf : 영상 분석 속도 최적화 |
| ✅ | test | 테스트 추가/수정 | ✅ test : 영상 분석 기능 단위 테스트 추가 |
| 🔧 | chore | 빌드, 설정, 패키지 관리 등 | 🔧 chore : ESLint 설정 업데이트 |
| 🎨 | design | UI/UX 디자인 변경 | 🎨 design : 분석 결과 페이지 디자인 개선 |
| 🔒 | security | 보안 관련 변경 | 🔒 security : 파일 업로드 보안 강화 |
| 🚀 | deploy | 배포 관련 | 🚀 deploy : 프로덕션 환경 배포 설정 |

### 커밋 메시지 작성 규칙

1. 이모지와 설명 사이에 콜론(`:`) 사용
2. 설명은 한국어로 작성
3. 설명은 간결하게 작성
4. 과거형 사용 금지 (예: "추가했다" ❌, "추가" ✅)
5. 마침표 사용 금지

## 🌿 브랜치 전략

### 브랜치 명명 규칙

| 브랜치 | 설명 |
| --- | --- |
| `main` | 프로덕션 브랜치 |
| `develop` | 개발 브랜치 |
| `feature/이슈번호-기능명` | 기능 개발 |
| `bugfix/이슈번호-버그명` | 버그 수정 |
| `hotfix/이슈번호-긴급수정` | 긴급 수정 |

### 브랜치 명명 예시

```
feature/123-video-analysis-pdf
bugfix/456-upload-error-fix
hotfix/789-critical-security-fix
```

### 브랜치 전략

1. 각 브랜치에서 개발 후 `main`으로 PR (릴리즈)
